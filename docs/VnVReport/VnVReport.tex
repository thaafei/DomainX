\section{Nonfunctional Requirements Evaluation}

This section reports verification results for the system's nonfunctional

\subsection{Performance \& Scalability Experiments}
\label{sec:nfr-perfscalability}

This subsection reports Revision~0 results for the performance and
scalability experiment specified in the VnV Plan (\S4.2.1).  The experiment
measures response times for dashboard load, filtering, AHP recomputation, and
export, and summarizes median and 95th-percentile (p95) values.

\subsubsection{Test Case Summary (T--NF1: Performance and Scalability)}
\begin{table}[H]
  \centering
  \caption{T--NF1 performance experiment: test case summary (Rev0).}
  \label{tab:tnf1-summary}
  \begin{tabular}{@{}p{0.22\textwidth}p{0.73\textwidth}@{}}
    \toprule
    \textbf{Test Case ID} &
      T--NF1 (Performance and Scalability) \\
    \textbf{Traceability} &
      PR-SL1 (Speed and Latency Requirement). \\
    \textbf{Initial State} &
      System deployed and accessible; existing domains available with known
      library counts (2 and 10). \\
    \textbf{Input} &
      Record time to (1) load the dashboard and render the ranking graph,

      Median and p95 response times are recorded; response times remain within

    \textbf{Actual Output} &

\subsubsection{Method (Rev0)}
Timings were collected manually using a stopwatch, measured from the moment
the action was initiated (click) to the moment the interface reached a stable
state (graph rendered or updated).  Due to manual timing, values include human
reaction time and should be treated as conservative upper bounds.  For small
sample sizes, p95 is reported conservatively as the maximum observed value.

\subsubsection{Results}
Table~\ref{tab:tnf1-results} summarizes the observed median and p95 response
times.

                  p{0.16\textwidth}@{}}
    \toprule
    \textbf{Action} & \textbf{Median (s)} & \textbf{p95 (s)} & \textbf{Notes} \\

      Time to render ranking view. \\
    Dashboard load (10 libraries, n=10) &

      Time to render ranking view. \\
    Recompute AHP (edit category weights, n=10) &
      3.61 & 3.90 &
      Time from save/apply to updated view. \\
    Apply domain filter &
      N/A & N/A &

    Export visualizations &
      $<0.5$ & $<0.5$ &
      Appeared instantaneous (below stopwatch resolution). \\
    \bottomrule
  \end{tabular}

\subsubsection{Analysis}
Across the measured actions, response times were within a few seconds, with
worst observed values below 4 seconds.  Dashboard load increased from a median
of 0.55 seconds (2 libraries) to a median of 2.65 seconds (10 libraries),
indicating higher latency as library count increases in the tested
configuration.  AHP recomputation completed in under 4 seconds in all trials,
which is acceptable for interactive use.

Domain filtering produced no observable narrowing of the domain list.  This is
treated as a functional defect and prevented measurement of filter latency in
Revision~0.

\subsubsection{Limitations and Next Steps}
Revision~0 results include only 2- and 10-library datasets and manual timing.
In the next revision, performance will be re-measured using the planned dataset
sizes (5, 10, 20, and 40 libraries) and automated timing collection where
possible.  The domain filtering defect will be addressed and re-tested to
capture filter latency.

\subsection{Usability}
\label{sec:nfr-usability}

This subsection reports Revision~0 results for the usability evaluation
(T--NF3).  The VnV Plan defines a one-to-five usability scale where higher
values indicate better usability and uses a mean rating $\geq 4.0$ as the
success criterion.  For Revision~0, available evidence consists of a pilot
task-based proxy evaluation (n = 1), reported below as preliminary evidence.

\subsubsection{Test Case Summary (T--NF3: Usability Pilot)}
\begin{table}[H]
  \centering
  \caption{T--NF3 usability pilot: test case summary (Rev0).}
  \label{tab:tnf3-summary}
  \begin{tabular}{@{}p{0.22\textwidth}p{0.73\textwidth}@{}}
    \toprule
    \textbf{Test Case ID} &
      T--NF3 (Usability) \\
    \textbf{Initial State} &
      System deployed and accessible; tester starts unauthenticated; no
      pre-existing domain required. \\
    \textbf{Input} &
      Tester completes eight core tasks (account creation/login, create domain,
      add metrics, add libraries, fill manual metrics, export data, visualize
      and export raw metrics, visualize and export AHP results) and records
      task ratings plus qualitative comments. \\
    \textbf{Expected Output} &
      Core tasks are discoverable and completable without developer assistance;
      charts and exports are understandable; required inputs are supported by
      clear feedback and error messaging (saving does not fail silently). \\
    \textbf{Actual Output} &
      All eight tasks were completed.  Two tasks showed reduced ease due to
      discoverability and feedback issues: adding metrics and locating manual
      metric fields. \\
    \textbf{Expected vs.\ Actual Match} &
      Partial (workflow completion succeeded; specific usability issues were
      identified). \\
    \bottomrule
  \end{tabular}
\end{table}

\subsubsection{Method (Rev0 Pilot)}
A proxy user (Ryan) completed a structured, asynchronous worksheet to reduce
developer influence.  The worksheet captured task \emph{difficulty} on a
one-to-five scale where 1 = Very Easy and 5 = Very Hard.

Because the VnV Plan convention is higher = better usability, we also report a
converted task \emph{ease} score using:
\[
  \text{EaseScore} = 6 - \text{DifficultyScore}.
\]
This conversion preserves the raw ratings while aligning the pilot results
with the plan's direction.

\subsubsection{Results}
Table~\ref{tab:usability-pilot} summarizes the proxy user's task ratings and
key observations.  Both the raw difficulty score and the converted ease score
are reported.

Summary statistics for the pilot:
\begin{itemize}
  \item Mean task difficulty: $3/2$ (1.5); median difficulty: 1.
  \item Mean converted ease: $9/2$ (4.5); median converted ease: 5.
  \item Lowest ease tasks (converted): add metrics; fill manual metrics (3/5).
\end{itemize}

\subsubsection{Analysis and Traceability}
The converted ease mean of 4.5/5.0 suggests the workflow is generally usable
in the pilot.  However, two targeted usability risks were identified:
(1) discoverability and feedback during metric creation, and
(2) navigational friction when locating manual metric fields among dense
inputs.  These issues map directly to the traceability targets for clear
feedback and efficient navigation.

Table~\ref{tab:usability-trace} summarizes traceability between usability
requirements and pilot evidence.

Additional improvement opportunities were noted by the proxy user, including
providing registration confirmation feedback and supporting bulk import of
libraries and automatic suggestion of repository URLs to reduce manual effort.

\subsubsection{Limitations and Next Steps}
The Revision~0 usability evidence is a pilot result with a single proxy user.
The Appendix~B usability survey with multiple participants will be executed in
the next revision to evaluate the plan's survey-based success criterion.
The pilot findings above will be converted into tracked usability issues and
re-tested after implementing improved discoverability for metric creation and
clearer save-time feedback for required fields.

\noindent \textbf{Evidence:} Pilot usability worksheet (Ryan), stored in the
project repository at \texttt{/test/reports/usability/UserTesting-Ryan.pdf}.

\subsubsection{Analysis and Traceability}
The existence of an automated test workflow integrated into pull requests
supports maintainability through testability and regression resistance.  The
workflow runs backend and frontend tests and produces a coverage report, which
enables repeatable inspection of covered and uncovered code paths.  The
presence of \texttt{test/README.md} further supports maintainability by
documenting how developers can run the test suite locally.

The observed overall backend coverage (73\%) does not meet the planned
$\geq 80\%$ target for Rev0.  This indicates that additional tests should be
added, particularly for modules with low coverage, to increase confidence in
future refactors and to align with MS-MR testability expectations.

This evaluation traces to MS-MR1--MS-MR6.  Rev0 evidence most directly
supports testability and documentation.  Explicit evidence for style
compliance via automated lint/format checks remains to be captured.

\subsubsection{Limitations and Next Steps}
In the next revision, maintainability will be strengthened by:
(1) increasing automated test coverage to reach the $\geq 80\%$ target,
(2) adding or capturing explicit lint/format checks in CI if required by the
VnV Plan, and
(3) recording a brief team code walkthrough note summarizing modular structure
and documentation completeness, with follow-up items tracked in GitHub issues.

\noindent \textbf{Evidence:}
GitHub Actions \texttt{tests.yml} workflow and latest successful run
(\texttt{backend-test} and \texttt{frontend-test}), including coverage output
(TOTAL coverage) and the \texttt{coverage-html} artifact; repository test
documentation at \texttt{/test/README.md}.
