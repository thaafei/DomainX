\section{Functional Requirements Evaluation}
\hspace{2em}This section presents the results of the functional tests defined in the project's
Verification and Validation (VnV) Plan. These tests verify that the implemented system
satisfies the functional requirements specified in the Software Requirements Specification
(SRS). The tests are organized according to the system's main feature areas.

\subsection{User Authentication and Profile Management}
\hspace{2em}The test results below verify that users can successfully authenticate
and manage their account information within the system. These tests ensure that
login, profile updates, and password modification operate correctly while
restricting access to authorized users only.

\subsubsection{Features Currently in Progress}

\hspace{2em}Some functional requirements described in the SRS are currently
under development or undergoing design revisions. These features are partially
implemented and are expected to be completed in future iterations of the system.
The current status of these requirements is summarized below.

\item \textbf{SR--AC1: Public Viewing and Restricted Editing}
The authorization mechanism required to allow public viewing while restricting
data modification to approved users is currently being implemented. Page-level
and button-level authorization controls are partially implemented and are still
being refined.

\item \textbf{FR2: User Self Sign-Up}
The original design included a public sign-up page. During development this
approach was reconsidered, and the system is being modified so that only a
Super Admin can create new user accounts. The self sign-up feature will therefore
be removed from the final implementation.


\item \textbf{Unauthorized API Access Control (SR--AC6, SR--AU1, SR--IM1)}

Unauthorized API access control is currently under development. The intended
behavior is that attempts to perform restricted actions (e.g., deleting a
domain without sufficient privileges) return an HTTP 403 Forbidden response
and generate an audit log entry. At present, restricted actions are hidden
in the user interface, while API-level authorization checks are still being
implemented and are not yet ready for testing.

Security requirements related to SQL injection protection and credential
storage are already satisfied through the use of Django ORM (which avoids
raw SQL queries) and environment variables for storing sensitive data.

\item \textbf{FR--4: Password Reset via Email}

The system is intended to allow users to reset their password through
an email verification workflow if they forget their credentials. The
expected behaviour is that the user receives a password reset email,
validates their email address using a token, and is able to set a new
password for future logins.

This functionality is currently under development and has not yet been
fully implemented or tested.
