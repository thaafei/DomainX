\subsection{Features Currently in Progress}

\hspace{2em}Some functional requirements identified in the SRS are still under development, while others were revised after further discussion and feedback. These items are not yet fully implemented or were removed from the final project scope. Their current status is summarized below.

\begin{description}

\item[\textbf{SR--AC1:}] \textbf{Public Viewing and Restricted Editing}

This requirement is currently under development as part of the system's authorization implementation. The intended behaviour is to allow public users to view published domain data while restricting modification privileges to approved users only. Page-level and button-level authorization controls have been partially implemented and are still being refined.

\item[\textbf{FR--2:}] \textbf{User Self Sign-Up}

This requirement was originally intended to allow users to create their own accounts through a public sign-up page. During development, this design was reconsidered, and the system is being revised so that only a Super Admin can create new user accounts. As a result, self sign-up will not appear in the final implementation.

\item[\textbf{SR--AC6, SR--AU1, SR--IM1:}] \textbf{Unauthorized API Access Control}

Unauthorized API access control is currently under development. The intended behaviour is that attempts to perform restricted actions, such as deleting a domain without sufficient privileges, will return an HTTP 403 Forbidden response and generate an audit log entry. At present, restricted actions are hidden in the user interface, while API-level authorization checks are still being implemented and are not yet ready for formal testing.

Security requirements related to SQL injection prevention and secure credential storage are already satisfied through the use of the Django ORM, which avoids raw SQL queries, and environment variables for sensitive configuration data.

\item[\textbf{FR--4:}] \textbf{Password Reset via Email}

This feature is intended to allow users to reset their password through an email verification workflow. The expected behaviour is that a user who forgets their password receives a reset email, verifies their identity through a tokenized link, and sets a new password for future logins.

This functionality is currently under development and has not yet been fully implemented or tested.

\item[\textbf{FR--9:}] \textbf{Bulk Data Upload}

This requirement was originally intended to support bulk upload of domain data through an Excel template. Based on supervisor feedback, this functionality was removed from the project scope and will not be implemented. Therefore, no testing was performed for this requirement.

\textbf{Relevant Functional Requirement(s):} FR--9, SR--INT1, UH--EU2

\item[\textbf{FR--6:}] \textbf{Publish Domain for Public Access}

This feature is currently under development as part of the remaining authorization work. The intended behaviour is that a completed domain can be published and made visible to all users, with its data available for public viewing and download. This functionality will be completed together with the remaining access-control features.

\textbf{Relevant Functional Requirement(s):} FR--6, OE--PR3, FR--14

\end{description}

\begin{enumerate}
  \item What went well while writing this deliverable?\\
  \textbf{Haniye:} One aspect that went well while writing this deliverable was that many of the test scenarios were already clear from the development process. While implementing features such as domain management, metric editing, and visualization, the system was frequently tested to verify correct behaviour. As a result, when preparing the VnV report it was easier to define the test inputs and expected outcomes because the implemented features had already been validated during development.
  \item What pain points did you experience during this deliverable, and how
    did you resolve them?\\
  \textbf{Haniye:} A minor challenge during this deliverable was ensuring that
    the tests described in the report aligned with the original VnV Plan while
    also reflecting the current implementation of the system. Some features had
    changed during development, which made it necessary to adjust how certain
    tests were documented. Feedback from the teaching assistant helped clarify
    how the tests should be structured, allowing the report to accurately
    represent the implemented functionality.
  \item Which parts of this document stemmed from speaking to your client(s) or
  a proxy (e.g. your peers)? Which ones were not, and why?\\
  \textbf{Haniye:} Some parts of this document were influenced by discussions
    with the project supervisor. For example, the bulk data upload feature was removed from the project scope after supervisor feedback
    indicated that it would not be necessary for the final tool. The
    supervisor also suggested removing the public sign-up feature and
    instead allowing a Super Admin to create user accounts directly, with
    users updating their password afterward.

  \item In what ways was the Verification and Validation (VnV) Plan different
  from the activities that were actually conducted for VnV?  If there were
  differences, what changes required the modification in the plan?  Why did
  these changes occur?  Would you be able to anticipate these changes in future
  projects?  If there weren't any differences, how was your team able to clearly
  predict a feasible amount of effort and the right tasks needed to build the
  evidence that demonstrates the required quality?  (It is expected that most
  teams will have had to deviate from their original VnV Plan.)\\
  \textbf{Haniye:} Most of the verification and validation activities followed
    the original VnV Plan. Some small differences occurred due to time
    constraints and changes in project priorities. For example, the bulk data
    upload feature was removed from the project scope based on supervisor
    feedback, so its related test was not implemented.
    Some tests were also influenced by issues discovered during deployment.
    For instance, while deploying the system, GitStats initially did not run
    correctly on the server and the analysis process continued for an
    unexpectedly long time. During debugging, the request timeout was reduced,
    which allowed us to observe the system’s failure message and verify that
    the application handled the error properly.

    In future projects, some of these adjustments could be anticipated by
    planning additional tests for integration and deployment scenarios.
\end{enumerate}
